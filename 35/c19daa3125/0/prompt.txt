# Org template config, admin dashboard & in-app config editor

## Context

Pawtograder creates per-assignment **handout** and **solution (grader)** repos by
generating from two **hardcoded** GitHub template repos:

- `assignment-create-handout-repo/index.ts` → `TEMPLATE_HANDOUT_REPO_NAME = "pawtograder/template-assignment-handout"`
- `assignment-create-solution-repo/index.ts` → `TEMPLATE_SOLUTION_REPO_NAME = "pawtograder/template-assignment-grader"`

Institutions that run their own GitHub org need their own templates. Today `github_org`
is just a `text` column on `classes`; there is **no `github_orgs` table** and no way to
configure templates per org or per class. Separately, the admin portal (`app/admin/*`)
can create/list/archive classes but gives admins no way to *operate inside* a course or
inspect per-org GitHub config, and there is no in-app way to edit the GitHub Actions
workflow files / `pawtograder.yml` that live in those repos (only a read-only
`repository-get-file` function exists).

This delivers four phases in one effort:

- **A.** Org-level handout/solution template defaults, overridable per class.
- **B.** Admin per-org dashboard showing each course's effective config.
- **C.** A Monaco-based editor for workflow & `pawtograder.yml` files with live
  validation that **commits back** to the repo.
- **D.** An admin dashboard course-context switcher to "act as an instructor".

Plus a rich set of Playwright e2e tests across all four.

## Phase A — Org & class template configuration

### Migration (`supabase/migrations/<ts>_github_org_templates.sql`)

1. **`github_orgs` table** keyed by org name (matches `classes.github_org`):
   - `org_name text primary key`
   - `default_handout_template_repo text not null default 'pawtograder/template-assignment-handout'`
   - `default_solution_template_repo text not null default 'pawtograder/template-assignment-grader'`
   - `created_at`, `updated_at` timestamptz, `created_by`, `updated_by` uuid → `auth.users`.
   - RLS: admins (global `user_roles.role='admin'`) full CRUD; instructors of any class
     in that org may `SELECT` (use the existing `authorizeforclassinstructor` pattern /
     an `EXISTS` against `classes` + `user_roles`). Follow the RLS style in
     `20250820185937_admin_portal_system.sql`.

2. **Class override columns** on `classes`:
   - `handout_template_repo text null`, `solution_template_repo text null`
   (NULL = inherit org default). No backfill needed.

3. **Resolution RPC** `resolve_class_template_repos(p_class_id bigint)` returning
   `(handout_template_repo text, solution_template_repo text)`:
   `COALESCE(class override, github_orgs default, hardcoded constant)`.
   `SECURITY DEFINER`, guarded by `authorizeforclassinstructor(p_class_id)`.

4. **Admin RPCs** (mirror `admin_get_classes` / `admin_create_class` in the admin-portal
   migration; all guard on global admin role):
   - `admin_get_github_orgs()` → one row per distinct org (union of `github_orgs` and
     orgs referenced by `classes.github_org`), with defaults and a `course_count`.
   - `admin_upsert_github_org(p_org_name text, p_handout text, p_solution text)`.
   - `admin_get_org_courses(p_org_name text)` → courses in the org with their override
     columns + resolved effective templates (call the resolution logic).

After writing the migration, run `npm run client-local` to regenerate
`utils/supabase/SupabaseTypes.d.ts` (auto-copied to `_shared/`). **Do not hand-edit it.**

### Edge functions — use resolved templates instead of constants

In `assignment-create-handout-repo/index.ts` and `assignment-create-solution-repo/index.ts`,
replace the hardcoded `TEMPLATE_*_REPO_NAME` passed to `createRepo(org, name, template, …)`
with a value resolved via the new RPC. Add a shared helper, e.g.
`resolveTemplateRepos(supabase, classId)` in `supabase/functions/_shared/GitHubSyncHelpers.ts`,
that calls `supabase.rpc("resolve_class_template_repos", { p_class_id })` and returns
`{ handout, solution }`, falling back to the existing constants if the row is missing.
Keep the constants as the documented fallback default.

## Phase D — Admin "act as instructor" course switcher

Admins already pass DB instructor authorization (`authorizeforclassinstructor` and
`assertUserIsInstructor` return true for `role='admin'`), but `ClassProfileProvider`
(`hooks/useClassProfiles.tsx`) shows "You don't have access to this course" when the admin
has no `user_roles` row for that class, and instructor UI gates on `role.role==='instructor'`.

**Approach (reliable, reuses the entire instructor surface):** idempotently provision the
admin a real instructor enrollment in the target class on entry.

1. **RPC `admin_enter_course_as_instructor(p_class_id bigint)`** (`SECURITY DEFINER`, guard
   global admin role): if the admin has no active `user_roles` row for `p_class_id`, create
   the required `profiles` (private + public) and an `instructor` `user_roles` row for them
   (follow profile/role creation already done in `admin_create_class` and the enrollment
   helpers). Idempotent. Audit-log via the existing audit mechanism used by other admin RPCs.
2. **Admin dashboard:** in `app/admin/AdminDashboardContent.tsx` (and the Phase B org-course
   table), add a **"Manage as instructor"** action per course that calls the RPC then
   navigates to `/course/[id]/manage`. Surface a course picker on `/admin` for quick access.
3. **Admin-viewing banner:** add a small client banner shown in the course manage area when
   the current user holds a global admin role (detect via `allOfMyRoles` from
   `useClassProfiles` / a lightweight `useIsAdmin` check). Mount it in
   `app/course/[course_id]/manage/loading.tsx`'s sibling — i.e. add it to the manage tree;
   if no manage `layout.tsx` exists, create
   `app/course/[course_id]/manage/layout.tsx` that renders the banner above `children`.

> Decision note for the implementer: provisioning a real instructor row is chosen over
> synthesizing a fake role because profile-dependent instructor actions (posting,
> assignments, audit author) would otherwise break. If polluting staff rosters is a
> concern, gate these rows out of staff listings with an `EXISTS` admin check rather than
> changing this approach.
